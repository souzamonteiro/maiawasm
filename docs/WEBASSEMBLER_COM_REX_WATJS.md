# Guide: Building a WebAssembler with WAT.js (REx)

This guide is **specifically** for a pipeline based on the parser generated by REx:

- grammar: [grammar/WAT.ebnf](../grammar/WAT.ebnf)
- generated parser: [WAT.js](../WAT.js)

It does not rely on the internal parser in [wat-assembler.js](../wat-assembler.js).

---

## 1) Target Architecture (Scalable)

Use 5 separate stages:

1. **Syntactic parse (REx / WAT.js)**
2. **Parse tree → semantic AST**
3. **Symbol/index resolution**
4. **WASM binary code generation**
5. **Fixture-based validation and testing**

Keeping these stages separate reduces coupling and makes feature expansion easier.

---

## 2) Recommended Folder Structure

Minimal suggestion:

- `src/rex/parse-wat.js` (wrapper around `WAT.js`)
- `src/rex/parse-tree-builder.js` (REx event handler)
- `src/ast/lower-wat-tree.js` (converts parse tree to semantic AST)
- `src/semantic/resolve-symbols.js`
- `src/codegen/emit-wasm.js`
- `src/codegen/binary-writer.js`
- `src/index.js` (`assembleFromWat(source)`)

Tests:

- `tests/fixtures/*.wat`
- `tests/expected/*.wasm` (when codegen is available)
- `tests/expected-ast/*.json` (already in place)

---

## 3) Stage 1 — Parse with WAT.js

Rebuild parser when grammar changes:

```bash
./build.sh
```

Then parse using:

- instantiate `new WAT(source, eventHandler)`
- call `parse_wat()`

Use the same pattern already implemented in [tests/test-parser.js](../tests/test-parser.js).

---

## 4) Stage 2 — Parse Tree → Semantic AST

The REx parse tree is excellent for syntax, but an assembler needs a stable semantic AST.

Define a simple model, for example:

- `ModuleNode { types, imports, funcs, tables, memories, globals, elems, datas, exports, start }`
- `FuncNode { id, params, results, locals, body }`
- `InstrNode { op, immediates }`

### Important rule

Semantic AST should be:

- independent from whitespace/comments
- independent from tokenization details
- stable for snapshot tests

---

## 5) Stage 3 — Symbol Resolution

Add a dedicated pass to resolve `$id` names into numeric indices:

- types
- functions
- tables
- memories
- globals
- labels

This pass is also the right place for structural validation:

- missing reference
- invalid index usage
- wrong entity type

---

## 6) Stage 4 — WASM Emission

With resolved semantic AST:

1. build module sections (type/import/function/table/memory/global/export/start/elem/data/code)
2. encode LEB128 values
3. emit opcodes + immediates
4. produce final byte vector

Always validate with:

- `WebAssembly.validate(bytes)`
- optional: `new WebAssembly.Module(bytes)`

---

## 7) Fixture-Based Test Strategy (Scalable)

### 7.1 Parser layer (already available)

- `node tests/test-parser.js`
- `AST_CHECK=1 node tests/test-parser.js`
- `ROUNDTRIP_CHECK=1 node tests/test-parser.js`

### 7.2 Semantic AST layer

Add tests for `lower-wat-tree`:

- input: `.wat` fixture
- output: `*.semantic-ast.json`
- snapshot comparison

### 7.3 Codegen layer

For each supported fixture:

- generate `.wasm`
- compare with expected output (or semantic validation)
- validate with `WebAssembly.validate`

---

## 8) Adding New Features with Low Overhead

For each new spec feature:

1. add one positive fixture + one negative fixture
2. ensure WAT.js parse coverage
3. update parse-tree → semantic-AST lowering
4. update resolution rules (if symbols are involved)
5. update codegen
6. add/update expected outputs

Checklist:

- [ ] parser accepts/rejects correctly
- [ ] semantic AST snapshot is correct
- [ ] syntax roundtrip is consistent
- [ ] generated WASM bytes are valid
- [ ] regression is covered

---

## 9) Recommended Implementation Order

1. Minimal viable subset: module + function + export + `i32.const` + `return`
2. Locals + numeric operations
3. Control flow (`block`/`loop`/`if`/`br`)
4. Memory + table
5. SIMD/GC/exceptions in small batches

This order keeps growth predictable and testable.

---
